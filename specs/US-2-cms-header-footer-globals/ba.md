## 1. Бизнес-цель (почему)

Сегодня «рамка сайта» — шапка и подвал — живёт в JSX (`Header.tsx`, `Footer.tsx`).
Каждая правка телефона, пункта меню, слогана или реквизита требует PR, ревью, сборки
и деплоя. Это ломает модель «CMS-first»: оператор декларируется владельцем контента,
но на деле зависит от разработки.

Задача выносит все редактируемые данные Header и Footer в Payload global
`SiteChrome` и превращает компоненты в тонкие server-component-читалки. После
закрытия US-2 оператор самостоятельно, из `/admin`, меняет телефон, меню, соцсети,
реквизиты, копирайтные ссылки — и изменения доезжают до prod за минуту без
деплоя. Это условие **публичного запуска** (в паре с US-1 seed).

Положительные эффекты второго порядка:

- Единое поле `phoneDisplay` / `phoneE164` делает Calltouch/CoMagic подмену номеров
  тривиальной задачей (позже).
- `Organization` JSON-LD (seo2) читает `telephone` и `sameAs` из того же global —
  единый источник правды, меньше рассинхронов.
- Вынос меню и соцсетей развязывает roadmap: programmatic-районы (неделя 7) и
  калькуляторы (неделя 3–4) смогут апдейтить навигацию без вмешательства в layout.

## 3. Цели и метрики

- **GOAL-1.** Операционная независимость оператора от разработки в правках рамки
  сайта.
  - Метрика: число правок Header/Footer, прошедших **без PR**, за календарную неделю
    после релиза.
  - Baseline: 0 (сейчас физически невозможно без кода).
  - Target: ≥ 80% всех правок рамки за первый месяц после релиза.

- **GOAL-2.** Скорость цикла правки «админка → prod».
  - Метрика: медианное время от `Save` в `/admin` до появления правки на prod.
  - Baseline: 10–25 мин (PR + CI + deploy).
  - Target: ≤ 60 сек (revalidation по tag + IndexNow pьего за url).

- **GOAL-3.** Ноль захардкоженных данных Header/Footer в компонентах (кроме разметки
  и логотипа бренда).
  - Метрика: `grep` по `Header.tsx` / `Footer.tsx` не находит ни одного номера
    телефона, ни одного пункта меню, ни одной соцссылки, ни одной строки реквизита
    в литералах.
  - Baseline: Header hard-coded 6 пунктов меню + 1 телефон + 1 CTA; Footer hard-coded
    5+5+5 ссылок + слоган + реквизитная строка.
  - Target: 0 литералов (допустимы только бренд-константы: `Обиход`, `®`, `LogoMark`).

- **GOAL-4.** Консистентность контактных каналов на всех страницах.
  - Метрика: на каждой публичной странице присутствуют кликабельные Telegram + MAX +
    WhatsApp + телефон (инвариант CLAUDE.md § Immutable).
  - Baseline: сейчас только телефон (placeholder) и `Telegram @obihod` нерабочей
    ссылкой.
  - Target: все 4 канала — из global, кликабельные deep-link, видимые на всех
    страницах (footer — минимум).

## 4. Требования (REQ-N.K)

### REQ-1.1 — Единый global `SiteChrome` с пятью секциями

- **Тип:** business rule + functional
- **Источник:** оператор (intake §8 ответ 1), PROJECT_CONTEXT §3
- **MoSCoW:** **Must**
- **Формулировка:** Система ДОЛЖНА предоставлять **один** Payload global с slug
  `site-chrome` и пятью группами полей (`header`, `footer`, `contacts`,
  `requisites`, `social`) ДЛЯ оператора как единая точка редактирования рамки сайта.
- **Обоснование:** в одном global оператор правит всё, что «не контент страниц».
  Два global (Header + Footer) удвоили бы когнитивную нагрузку и породили бы
  рассинхроны полей `telephone` / соцсетей, которые нужны и сверху, и снизу.
- **Зависимости:** REQ-1.2 … REQ-1.6 (все живут внутри `SiteChrome`).
- **Открытые вопросы:** финальное имя (`site-chrome` vs `chrome` vs `navigation`) —
  уточнить у `sa` (не влияет на бизнес-смысл).

### REQ-1.2 — Раздельные поля `phoneDisplay` и `phoneE164`

- **Тип:** functional
- **Источник:** оператор (intake §8 ответ 2)
- **MoSCoW:** **Must**
- **Формулировка:** Система ДОЛЖНА хранить телефон в двух независимых полях —
  `phoneDisplay` (строка формата `+7 (985) 170-51-11`, для рендера) и
  `phoneE164` (строка формата `+79851705111`, для `tel:`-ссылок и JSON-LD) — ДЛЯ
  корректного отображения человеком и машинной обработки одной и той же сущности.
- **Обоснование:** форматированный номер плохо попадает в `tel:`, а E.164 некрасив
  для глаза. Разделение снимает конфликт и делает подключение Calltouch подмены
  номеров (roadmap неделя 5–6) предсказуемым — меняется только `phoneDisplay` /
  `phoneE164` в global.
- **Зависимости:** REQ-1.1, REQ-3.1 (UI читает оба), REQ-4.1 (SEO читает E.164).
- **Открытые вопросы:** валидация формата E.164 в Payload (regex `^\+7\d{10}$`) —
  детали за `sa`. Seed-значение на старте: `phoneDisplay = "+7 (985) 170-51-11"`,
  `phoneE164 = "+79851705111"`.

### REQ-1.3 — Массив соцсетей с enum-типом и drag-and-drop сортировкой

- **Тип:** functional
- **Источник:** оператор (intake §8 ответ 3)
- **MoSCoW:** **Must**
- **Формулировка:** Система ДОЛЖНА хранить соцсети как массив объектов вида
  `{ type, url }`, где `type` ограничен enum `telegram | max | whatsapp | vk |
  youtube | yandex-zen | other`, а порядок элементов управляется drag-and-drop в
  `/admin`, ДЛЯ того чтобы оператор мог добавлять/удалять/переставлять каналы без
  разработки и чтобы UI / JSON-LD могли рендерить иконку по типу.
- **Обоснование:** фиксированные слоты (`telegramUrl`, `maxUrl`, `whatsappUrl`)
  ломают расширение под новые каналы. Массив + enum — гибкость + типобезопасность.
- **Зависимости:** REQ-1.1, REQ-3.2 (UI), REQ-4.1 (JSON-LD `sameAs`).
- **Открытые вопросы:** нужно ли поле `label` для подписи в футере (сейчас hard-code
  «Telegram @obihod», «ВКонтакте») — рекомендация: подпись рендерится по `type`,
  дополнительного поля не нужно. Уточнить у `ui` при передаче макета.

### REQ-1.4 — Меню как union: anchor / route / external с валидацией

- **Тип:** functional + business rule
- **Источник:** оператор (intake §8 ответ 4)
- **MoSCoW:** **Must**
- **Формулировка:** Система ДОЛЖНА хранить пункты меню (header и footer-колонок)
  как массив объектов с дискриминатором `kind`:
  - `anchor` → поле `anchor` (строка без `#`, рендер `#<anchor>` на текущей странице);
  - `route` → поле `route` (относительный путь, начинается с `/`, рендер через
    Next `<Link>`);
  - `external` → поле `url` (полный URL с `http(s)://`, рендер `<a target="_blank"
    rel="noopener noreferrer">`);
  и Payload ДОЛЖЕН валидировать корректность поля по выбранному `kind` ДЛЯ
  защиты оператора от невалидных ссылок и корректного SEO/UX.
- **Обоснование:** сейчас в Header все ссылки — anchor’ы, которые ломаются на
  страницах без соответствующей секции (районные, Services). После seed появятся
  реальные роуты. Свободный `href`-текст оставит дыру для опечаток и `javascript:`
  вставок.
- **Зависимости:** REQ-1.1, REQ-3.3 (UI-рендер), REQ-2.2 (деградация на пустых).
- **Открытые вопросы:** максимум пунктов в header (UI ограничивает визуально — 6–7)
  — за `ui`. Валидация `route` на существование страницы — вне scope, ответственность
  оператора.

### REQ-1.5 — Реквизиты: временный ИНН + пустые поля с подсказкой

- **Тип:** business rule + functional
- **Источник:** оператор (intake §8 ответ 5), memory `project_inn_temp.md`
- **MoSCoW:** **Must**
- **Формулировка:** Система ДОЛЖНА хранить в секции `requisites` поля: `legalName`
  (строка, пустое с подсказкой), `inn` (строка, seed-значение `7847729123` с
  подсказкой «временный ИНН, подлежит замене при регистрации юрлица Обиход»),
  `kpp` / `ogrn` / `legalAddress` (пустые строки с подсказкой «заполнить после
  регистрации юрлица») ДЛЯ прозрачной передачи B2B-клиентам актуальных данных и
  для явной маркировки в админке, что значение временное.
- **Обоснование:** B2B-сегмент (УК, ТСЖ, госзаказ) проверяет контрагента перед
  договором — пустые или placeholder-реквизиты режут конверсию. Временный ИНН
  лучше, чем ничего; но обязательно с маркером в UI админки, чтобы оператор не
  забыл поменять.
- **Зависимости:** REQ-1.1, REQ-3.4 (рендер в footer-bottom), REQ-4.1 (`taxId` в
  `Organization` JSON-LD).
- **Открытые вопросы:** дублировать ли `inn` / `legalAddress` из `SeoSettings.organization`
  или сделать `SiteChrome.requisites` источником правды, а `SeoSettings.organization`
  читать оттуда — **это архитектурный вопрос к `sa` + `dba`**, ключевой для
  предотвращения рассинхронов.

### REQ-1.6 — Копирайт: авто-год + ссылки из global

- **Тип:** functional + business rule
- **Источник:** оператор (intake §8 ответ 8)
- **MoSCoW:** **Must**
- **Формулировка:** Система ДОЛЖНА рендерить копирайтную строку в footer как
  `© <new Date().getFullYear()> <legalName или бренд «Обиход»> · <privacyLink>
  · <ofertaLink>`, где `privacyUrl` и `ofertaUrl` — строковые поля в
  `SiteChrome.footer`, а ИНН **не включается** в эту строку (он уже в блоке
  реквизитов), ДЛЯ того чтобы копирайт не ломался при смене года и оператор мог
  поменять целевые страницы Политики / Оферты без кода.
- **Обоснование:** в текущем `Footer.tsx` `© 2026` захардкожен — в 2027 году
  вернётся регрессия. ИНН в копирайте дублирует блок реквизитов и тянет в каждую
  правку.
- **Зависимости:** REQ-1.1, REQ-1.5 (legalName/brand), REQ-3.4 (рендер).
- **Открытые вопросы:** дефолтные значения ссылок — `/politika-konfidentsialnosti/`
  и `/oferta/` (как в текущем JSX). Подтвердить у `po`.

### REQ-1.7 — Каналы клиента Telegram + MAX + WhatsApp + телефон обязательно видимы и кликабельны

- **Тип:** business rule (инвариант проекта)
- **Источник:** CLAUDE.md § Immutable → «Каналы коммуникации с клиентом»,
  PROJECT_CONTEXT §2
- **MoSCoW:** **Must**
- **Формулировка:** На каждой публичной странице сайта ДОЛЖНЫ одновременно
  присутствовать кликабельные контакты через **Telegram**, **MAX**, **WhatsApp**
  и **телефон** — как минимум в Footer; если соответствующая ссылка не заполнена
  в `SiteChrome.social[]` или `SiteChrome.contacts.phoneE164` — Система ДОЛЖНА
  писать предупреждение в Payload admin-UI (missing critical channel) ДЛЯ
  соответствия инварианту проекта.
- **Обоснование:** это закреплённый корневой инвариант. Потерять любой из четырёх
  каналов — нарушить стратегию «удобно клиенту, как ему удобно»; для старшей B2C
  аудитории WhatsApp критичен, для B2B — телефон, для younger — Telegram, для
  РФ-alt — MAX.
- **Зависимости:** REQ-1.2 (phoneE164), REQ-1.3 (social[]).
- **Открытые вопросы:** форма уведомления (toast в админке vs блокирующая
  валидация) — за `sa` + `ui`. Рекомендация BA: **warning, а не error** — оператор
  временно может работать без одного канала (например, MAX ещё не подключён).

### REQ-2.1 — Revalidation ≤ 60 сек после сохранения global

- **Тип:** non-functional (performance) + functional
- **Источник:** intake §9 «Метрика успеха» (≤ 60 сек), GOAL-2
- **MoSCoW:** **Must**
- **Формулировка:** При сохранении `SiteChrome` в `/admin` Система ДОЛЖНА в течение
  ≤ 60 секунд инвалидировать кеш всех публичных страниц и повторно отрендерить
  Header/Footer с новыми данными ДЛЯ соблюдения SLA «правка доезжает до prod за
  минуту без деплоя».
- **Обоснование:** `afterChange` Payload-hook → вызов существующего `/api/revalidate`
  с `tag=site-chrome` + `revalidatePath('/', 'layout')`; паттерн уже применяется
  для Services (см. `site/app/api/revalidate/route.ts`).
- **Зависимости:** REQ-1.1, инфраструктура RSC-кеша Next 16.
- **Открытые вопросы:** выбор между `revalidateTag('site-chrome')` и
  `revalidatePath('/', 'layout')` — за `sa` + `tamd`. Рекомендация BA:
  **оба** — tag для точечной инвалидации, path-layout для надёжности на первом
  запуске.

### REQ-2.2 — Мягкая деградация при пустом / отсутствующем global

- **Тип:** non-functional (reliability)
- **Источник:** intake §9 риски
- **MoSCoW:** **Must**
- **Формулировка:** Если `SiteChrome` пустой, не создан или чтение из Payload
  падает с ошибкой, Система ДОЛЖНА отрендерить Header и Footer с дефолтными
  значениями (бренд «Обиход», пустое меню или минимальный набор anchor-ов,
  копирайт с текущим годом, без реквизитов) и НЕ ДОЛЖНА показывать белый экран
  или 500 ДЛЯ того, чтобы первый запуск и временный сбой БД не убивали сайт.
- **Обоснование:** global до первого seed — норма. Payload readFailure на prod —
  реальный риск (БД пустая, пинг падает). Белый экран = нулевая конверсия.
- **Зависимости:** REQ-2.1, fallback-константы в `site/components/marketing/`.
- **Открытые вопросы:** где хранить fallback (JSON-seed рядом с компонентом vs
  inline в RSC) — за `fe1/fe2`.

### REQ-2.3 — Access: публичное чтение, запись только admin

- **Тип:** business rule (security)
- **Источник:** intake §9 риски
- **MoSCoW:** **Must**
- **Формулировка:** Система ДОЛЖНА разрешать `read` для `SiteChrome` публично
  (`read: () => true`), а `update` — только пользователям с ролью `admin` ДЛЯ
  защиты редактирования рамки от неавторизованного доступа и одновременной
  возможности RSC-чтения без авторизации.
- **Обоснование:** Header/Footer рендерятся RSC'ом в публичном layout, значит
  read должен быть неаутентифицированным. Update — чувствительное действие: меняет
  все страницы разом.
- **Зависимости:** REQ-1.1, текущая модель `Users` с role `admin` / `manager`.
- **Открытые вопросы:** достаточно ли `admin`, или `manager` тоже? Рекомендация BA:
  **только admin** — footer-bottom содержит юридические реквизиты, ошибка стоит
  дороже, чем удобство.

### REQ-2.4 — Max-length и валидация длины строк

- **Тип:** non-functional (UX)
- **Источник:** intake §9 риски, консультация `ui`
- **MoSCoW:** **Should**
- **Формулировка:** Текстовые поля `SiteChrome` ДОЛЖНЫ иметь лимиты длины в схеме
  Payload (maxLength: title меню — 20, phoneDisplay — 20, подпись соцсети/type —
  enum, slogan — 160, legalName — 120, labels CTA — 24) ДЛЯ того чтобы оператор
  не ломал макет длинной строкой.
- **Обоснование:** макет Header строго-сеточный, длинный пункт разваливает
  layout на мобиле; footer-slogan с rich-text может разрастись.
- **Зависимости:** REQ-1.1, передача конкретных чисел из макетов `ui`.
- **Открытые вопросы:** конкретные лимиты — за `ui` + `sa` (на ревью схемы).

### REQ-4.1 — SEO: `Organization` JSON-LD читает `phoneE164` + `sameAs` из `SiteChrome`

- **Тип:** functional (интеграция с SEO-слоем)
- **Источник:** intake §9 состав команды → `seo2`
- **MoSCoW:** **Must**
- **Формулировка:** `Organization` / `LocalBusiness` JSON-LD на сайте ДОЛЖЕН брать
  `telephone` из `SiteChrome.contacts.phoneE164` и `sameAs[]` из
  `SiteChrome.social[].url` (единый источник правды), а не из отдельных полей
  `SeoSettings.organization.telephone` и `SeoSettings.sameAs` ДЛЯ предотвращения
  рассинхронов между UI и структурированной разметкой.
- **Обоснование:** сейчас в `SeoSettings` дублирующие поля `telephone` и `sameAs`,
  которые будут разъезжаться с UI при следующей же правке.
- **Зависимости:** REQ-1.2, REQ-1.3, миграция/передача владения в `SeoSettings`
  (или read-through).
- **Открытые вопросы:** удалить ли `SeoSettings.organization.telephone` и
  `SeoSettings.sameAs` или оставить как override — **ключевой архитектурный вопрос
  к `sa`**. Рекомендация BA: оставить `SeoSettings` источником для ИНН/legalName
  (до решения по миграции), но `telephone` и `sameAs` **мигрировать в `SiteChrome`**

## 7. Риски и допущения

**Допущения:**

- A1. Payload `afterChange` глобалов стабильно вызывается (проверено на `Services`).
- A2. Существующий `/api/revalidate` принимает произвольные `tag` — проверено в
  исходнике, `tag=site-chrome` сработает без доработки эндпоинта.
- A3. Есть 1 активный `admin`-пользователь у оператора.
- A4. Операторский браузер поддерживает drag-and-drop Payload (стандартная фича
  Payload 3).

**Риски:**

- R1. **Пустой global на первом запуске prod.** Если seed не прогнали — Header
  и Footer не должны сломаться (REQ-2.2). Митигация: fallback-константы в
  компонентах + seed в релизном плане + e2e-тест с пустой БД (`qa1/qa2`).
- R2. **Неконсистентные href в меню от оператора.** Оператор выбирает `route` и
  пишет `uslugi/` без ведущего `/`. Митигация: Payload-валидация в REQ-1.4 +
  подсказка под полем + автоматическое добавление `/` в рантайме (fallback).
- R3. **Рассинхрон `phoneE164` / `phoneDisplay`.** Оператор меняет один без другого.
  Митигация: при сохранении `phoneE164` Payload-hook проверяет, что `phoneDisplay`
  форматно соответствует (или хотя бы обновлён в той же транзакции); в админке —
  warning. Финальная логика — за `sa`.
- R4. **SSR-кеш RSC и Turbopack держат старый global.** Даже после
  `revalidatePath('/', 'layout')` прокси-CDN Beget может кэшировать. Митигация:
  `revalidateTag('site-chrome')` + `Cache-Control: s-maxage=60, stale-while-revalidate`
  на layout + проверка после релиза (pa / aemd отслеживает TTFB).
- R5. **Ограничение длины строк vs layout.** Длинный пункт меню ломает Header на
  mobile. Митигация: REQ-2.4 + визуальный regress в Playwright.
- R6. **Рассинхрон с `SeoSettings`.** Сейчас в `SeoSettings.organization.telephone`
  и `SeoSettings.sameAs` лежат дубли. Митигация: REQ-4.1 + `sa` финализирует
  миграцию в спеке. Риск приоритетный — `seo2` и `cw` в курсе.
- R7. **Оператор забудет заполнить временный ИНН.** Митигация: явная подсказка
  в админке (REQ-1.5) + pending-задача у `po` на замену ИНН после регистрации
  юрлица.
- R8. **Access-регрессия.** `read: () => true` может по ошибке оказаться у `update`.
  Митигация: unit-тест на access + `cr` внимательно смотрит.

## 8. Acceptance Criteria (BA-уровень)

Критерии формулируются с точки зрения оператора — «что я должен увидеть, чтобы
сказать “приняли”». Детализация Given/When/Then — зона `sa`.

- **AC-BA-1.** В `/admin` есть пункт `SiteChrome` (или эквивалентное имя) с
  пятью секциями `header`, `footer`, `contacts`, `requisites`, `social`, и все поля
  из REQ-1.1…REQ-1.6 присутствуют.
- **AC-BA-2.** Редактирование любого поля `SiteChrome` и сохранение в `/admin`
  не требует от оператора открывать терминал, git или DevTools — всё через UI.
- **AC-BA-3.** После `Save` в `/admin` изменения на prod появляются за ≤ 60 секунд
  без деплоя (проверяется таймером + hard-reload).
- **AC-BA-4.** Файлы `site/components/marketing/Header.tsx` и `Footer.tsx` не
  содержат ни одного номера телефона, ни одного пункта меню, ни одной соцссылки,
  ни одной строки реквизитов в литералах — только JSX-разметку, импорт LogoMark и
  чтение из global (проверяется grep'ом и `cr`).
- **AC-BA-5.** На каждой публичной странице сайта (главная + любая programmatic
  посадка в будущем) в Footer присутствуют все четыре канала — Telegram, MAX,
  WhatsApp, телефон — как кликабельные ссылки; если любой из них не заполнен в
  global, админка показывает warning (не блокирующую ошибку).
- **AC-BA-6.** При пустом / отсутствующем `SiteChrome` (или при сбое Payload-чтения)
  главная страница возвращается со статусом 200 и отрендеренной рамкой по дефолту
  (бренд + дефолтный копирайт + пустое меню), без белого экрана и без 500.
- **AC-BA-7.** Неавторизованный запрос к чтению данных `SiteChrome` возвращает
  поля (так как `read: () => true`); попытка записи без роли `admin` возвращает
  403/401.
- **AC-BA-8.** `view-source` любой публичной страницы содержит `Organization`
  JSON-LD, где `telephone` совпадает с `phoneE164` из `SiteChrome.contacts`, а
  `sameAs[]` совпадает с `social[].url` из `SiteChrome.social`.
- **AC-BA-9.** Попытка ввести в поле `nav.title` строку длиннее лимита приводит к
  ошибке валидации в `/admin` (не обрезке без предупреждения).
- **AC-BA-10.** В `/admin` **нет** пункта «сайт-wide баннер» и **нет** region
  switcher'a (подтверждение out-of-scope).
