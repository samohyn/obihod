## Резюме запроса

Сделать **Payload admin (`/admin`)** продуктовым инструментом контент-менеджера, а не дефолтной админкой Payload со служебными названиями полей. Цель — оператор открывает админку и за 30 секунд понимает, как добавить район, завести новую услугу, обновить цену, опубликовать кейс, поменять баннер на главной. Сейчас — не понимает.

Работа делится на два слоя:

1. **UX/визуал/IA админки** — навигация, группировка коллекций, подписи полей, подсказки, порядок полей, collapsibles, дашборд, brand-логика (BrandLogo, палитра, типографика). Ведущие роли: `ux` → `ui` → `art`.
2. **Функциональные правки Payload** — custom components (preview, duplicate-to-district, bulk-actions), расширение `admin.description`, `admin.useAsTitle`, `admin.defaultColumns`, сторонние Lexical-плагины (если нужны), ревизии/drafts, access-валидация. Ведущая роль: `be4`.

**Без этой задачи:** оператор не наполняет сайт → US-4..US-10 заблокированы логически (некуда складывать результаты семантики и контента).

## Тип задачи

- **Тип:** design-refresh (UX/визуал) + feature (кастомные компоненты + поля).
- **Срочность:** `high` (P0 по программе).
- **Сложность (грубая оценка in, верифицирует sa):** 2–3 недели полного цикла UX-исследование → макеты → имплементация → приёмка.

## Контекст (что уже есть)

- Payload 3, embed в Next.js 16 App Router, prod на `/admin`.
- Частичное брендирование (US-2 остатки): кремовый фон, BrandLogo, группы навигации `01·Заявки / 02·Контент / 03·Медиа / 04·SEO / 05·Рамка сайта / 09·Система`.
- Фактические коллекции (через be4): `Services`, `Districts`, `ServiceDistricts`, `Cases`, `Blog`, `B2BPages`, `Persons`, `Leads`, `Redirects`, `Media`, `Users` + globals `SeoSettings`, `SiteChrome`.
- Prices и FAQ встроены в `Services` (не отдельные коллекции — расхождение с CLAUDE.md фиксируется в US-5).
- `site/app/(payload)/admin/importMap.js` коммитится в репо, ломается при добавлении новых global/collection без `pnpm generate:importmap` перед build (learnings 2026-04-23).
- Drafts/versions уже включены на всех публичных коллекциях (be4).
- SiteChrome — 405 строк конфигурации, длинный скролл без tabs (be4 называет это UX-долгом).

## Открытые вопросы к оператору (до передачи → ba)

1. **Топ-5 раздражителей в текущей админке.** Дай список конкретных мест где тебя бесит: подпись поля, длина формы, отсутствие подсказок, поиск, сортировка, мобильная версия, preview, загрузка медиа. Нужно для UX-исследования (ux + re).
2. **Референсы админок, которые нравятся.** Sanity Studio, Contentful, Strapi, Directus, Webflow CMS, Tilda CMS, WordPress Gutenberg, Notion, Airtable — что-то из этого вдохновляет? Можно скриншоты или ссылки.
3. **Роли доступа в админке.** Сейчас один тип `admin`. Нужны ли роли: `content-manager` (без settings), `seo-manager` (только SEO-поля), `operator-leads` (только Leads, read-only), `developer` (всё)? Это влияет на схему access и дизайн side-nav (разным ролям — разные пункты меню).
4. **Preview контента.** Когда правишь страницу в админке, хочешь видеть «как это будет выглядеть» до публикации: (а) live-preview в iframe рядом с формой, (б) кнопка «Открыть preview» в новой вкладке, (в) достаточно publish → refresh prod?
5. **Mobile-доступ к админке.** Ты правишь контент с телефона/планшета или только с десктопа? От этого зависит scope UI-редизайна.

## Ответы оператора (зафиксировано 2026-04-24)

1. **Топ-5 раздражителей — один главный: «я не понимаю что там нужно делать тупо».** Боль не в визуале, а в **отсутствии on-screen guidance** — непонятно куда кликать, что за поле, зачем, что из этого обязательно, какой типичный сценарий. **Прямой намёк оператора:** «может быть решается отдельном файлом с понятной инструкцией».

   _Интерпретация для ba/ux:_ ключ US-3 — встроенная документация-инструкция в интерфейсе админки, а не переделка UX Payload как такового. Рассмотреть: (а) Dashboard-tile «Как пользоваться» со списком частых сценариев, (б) inline help-блоки на каждом collection-edit view, (в) подсказки `admin.description` на каждом поле (сейчас их почти нет), (г) стартовая страница админки со списком операций «Добавить район / Обновить цену / Опубликовать кейс / Заменить баннер» в формате walkthrough, (д) Markdown/MDX-справочник в `/admin/docs/` как отдельная страница Payload (если это возможно в Payload 3).

2. **Референс — Notion.** Берём от Notion: левая sidebar-навигация, мягкая типографика, большие поля с контекстными подсказками, блочная модель контента, минималистичная визуальная плотность, hover-hints, микро-анимации. **Не берём:** базовую модель данных (у нас реляционные коллекции, а не документы).

3. **Роли доступа — единый `admin`.** ACL не усложняем. Вся навигация и формы проектируются под одну роль. Снимает значительную часть complexity (side-nav filtering, conditional fields, access rules).

4. **Preview — да, делаем.** Должна быть возможность увидеть страницу до публикации. `sa`/`be4` выбирают между (а) live-iframe preview внутри формы, (б) кнопка «Открыть preview» → новая вкладка с `?preview=...&token=...`, (в) Payload 3 draftMode + `/api/preview` + `/api/exit-preview`.

5. **Только desktop.** Мобильный админ-UI — out of scope. Экономия на адаптиве, фокус на качестве desktop-экспириенса.

**Scope US-3 финализирован. Ключевой сдвиг фокуса:** это не «редизайн админки», а «**добавить слой документации/guidance в админку + минимальные UX-правки + preview**». Значительно меньше scope, чем изначально предполагалось.
