## 1. Бизнес-цель (почему)

Prod-инстанс сайта `obikhod.ru` развёрнут и технически здоров
(`/api/health?deep=1` отдаёт 200), но БД пуста: в Payload нет ни одной записи
`Services`, `Districts`, `ServiceDistricts`, `Cases`, `Persons`, `SeoSettings`
не наполнен. Вся публичная витрина — включая хедер «Услуги» и programmatic-роуты
вида `/arboristika/odincovo/` — возвращает 404 и недоступна для индексации в
Яндекс.Вебмастере. Пока БД пустая, **любая** маркетинговая и SEO-активность
бессмысленна: трафик приходит на 404, платный клик в Директе сливается, а
бригадир не получает заявок. Закрытие US-1 снимает блокер публичного запуска и
открывает следующие фазы roadmap — подключение форм/калькуляторов, настройку
амоCRM-вебхуков, подачу sitemap на индексацию.

## 3. Цели и метрики

- **GOAL-1 — Разблокировать публичный запуск пилотной географии.**
  - Метрика: доля HTTP 200 среди 28 programmatic-URL `/<service-slug>/<district-slug>/`
    (4 кластера × 7 пилотных районов).
  - Baseline: 0 / 28 (100% — 404).
  - Target: 28 / 28 = 100%, всё по HTTP 200 (draft-статус внутри Payload допустим,
    но рендер страницы и попадание в `sitemap.xml` — обязательно).

- **GOAL-2 — Обеспечить SEO-минимум для индексации Яндексом.**
  - Метрика: доля LandingPages с заполненными `metaTitle` (≤60), `metaDescription`
    (140–160), `h1`, `leadParagraph` из `ServiceDistricts`.
  - Baseline: 0%.
  - Target: 100% из 28 LP + корневые Services (4 шт.) + Districts (7 шт.).

- **GOAL-3 — Сохранить доверие к бренду при первом контакте.**
  - Метрика: 0 выявленных анти-TOV формулировок на публичных страницах (проверка
    `qa` + `cw` по стоп-словам из PROJECT_CONTEXT.md §2 и CLAUDE.md Immutable).
  - Baseline: N/A (нет страниц).
  - Target: 0 совпадений по словарю запрещённых формулировок.

- **GOAL-4 — Seed стать переиспользуемым активом команды.**
  - Метрика: повторный запуск `pnpm seed` на той же БД не создаёт дублей и не
    падает по ошибкам уникальности.
  - Baseline: скрипт `site/scripts/seed.ts` идемпотентен по slug, но покрывает
    1 услугу × 2 района, без полноты MVP.
  - Target: скрипт покрывает 4 кластера × 7 районов, повторный запуск — 0
    изменений + exit 0.

## 4. Требования (REQ-N.K)

### REQ-1.1 — Идемпотентность seed-скрипта

- Тип: non-functional (reliability)
- Источник: intake §«Риски / зависимости» п.1; PROJECT_CONTEXT.md §7 блокер;
  существующая реализация `site/scripts/seed.ts` (см. helper `findOneBySlug`).
- MoSCoW: **Must**
- Формулировка: Система ДОЛЖНА при повторном запуске `pnpm seed` на
  непустой БД не создавать дубликатов и не падать ДЛЯ `do` и `dba`, чтобы
  операция seed была безопасна при инциденте / миграции / refresh staging.
- Обоснование: одноразовый seed в prod — источник рисков (человеческий фактор,
  второй деплой по ошибке). Без идемпотентности блокируем сценарий refresh.
- Зависимости: REQ-1.2 (полнота объёма).
- Открытые вопросы: нет.

### REQ-1.2 — Полнота объёма MVP: 4 кластера × 7 районов

- Тип: functional
- Источник: intake §«Ответы оператора 2026-04-23» п.2;
  PROJECT_CONTEXT.md §4 (каталог услуг) + §5 (пилотная география).
- MoSCoW: **Must**
- Формулировка: Система ДОЛЖНА после прогона seed содержать ровно 4 опубликованные
  корневые услуги (`Services`: `arboristika`, `ochistka-krysh`, `vyvoz-musora`,
  `demontazh`), 7 районов (`Districts`: `odincovo`, `krasnogorsk`, `mytishchi`,
  `khimki`, `istra`, `pushkino`, `ramenskoye`) и 28 записей `ServiceDistricts`
  (полный декартов квадрат 4 × 7) ДЛЯ seo1/seo2, чтобы все 28 publicных URL
  отвечали HTTP 200 и попадали в sitemap.
- Обоснование: MVP-квадрат зафиксирован оператором; меньший объём не проходит
  порог «живого» сайта, больший — за пределами scope.
- Зависимости: REQ-1.1, REQ-1.3, REQ-1.4, REQ-2.1.
- Открытые вопросы: slug районов — финализировать в `sa` (латиница kebab-case,
  соответствует регулярке `^[a-z][a-z0-9-]{2,40}$` из `Services.ts`).

### REQ-1.3 — SEO-минимум для каждой корневой услуги и LandingPage

- Тип: functional
- Источник: `site/collections/Services.ts` (обязательные `metaTitle ≤60`,
  `metaDescription 140–160`, `h1 ≤90`, `intro richText`);
  `site/collections/Districts.ts`; `site/collections/ServiceDistricts.ts`
  (обязательный `leadParagraph`); CLAUDE.md §SEO.
- MoSCoW: **Must**
- Формулировка: Система ДОЛЖНА для каждой из 4 `Services` и 7 `Districts`
  содержать непустые поля `metaTitle`, `metaDescription`, `h1`/`nameNominative`,
  а для каждой из 28 `ServiceDistricts` — непустой `leadParagraph`, ДЛЯ seo2 и
  qa, чтобы `generateMetadata` возвращал корректные мета-теги и сайт был
  готов к подаче в Яндекс.Вебмастер.
- Обоснование: отсутствие `metaDescription` валит Payload-валидацию (minLength
  140). Пустой `leadParagraph` — то же самое. Без мета индексация даёт нулевой
  результат.
- Зависимости: REQ-1.2, REQ-2.3.
- Открытые вопросы: будет ли seed создавать `ServiceDistricts` в статусе `draft`
  (с `noindexUntilCase=true`, как в текущем seed.ts), `review` или `published` —
  решение `sa` + `seo1` (см. §9).

### REQ-1.4 — Падежные формы названий районов

- Тип: business rule
- Источник: `site/collections/Districts.ts` (обязательные `nameNominative`,
  `namePrepositional`, `nameDative`, `nameGenitive`).
- MoSCoW: **Must**
- Формулировка: Система ДОЛЖНА для каждого района в seed содержать 4 падежные
  формы (именительный, предложный, дательный, родительный) ДЛЯ `cw` и seo2,
  чтобы шаблоны фраз «спил деревьев **в Одинцово**», «выезд бригады **по
  Одинцово**» рендерились грамматически корректно.
- Обоснование: Payload сделал эти поля required именно из-за programmatic-SEO.
  Поставить пустые строки = падение валидации.
- Зависимости: REQ-1.2.
- Открытые вопросы: нет — формы берём из справочника `seo1` или из `cw`;
  конкретные значения фиксирует `sa` в AC.

### REQ-1.5 — Каналы коммуникации: Telegram + MAX + WhatsApp + телефон

- Тип: business rule (invariant)
- Источник: CLAUDE.md §Immutable (каналы); PROJECT_CONTEXT.md §2.
- MoSCoW: **Must**
- Формулировка: Система ДОЛЖНА в seed-значениях (SeoSettings, `Persons.sameAs`,
  тексты `leadParagraph`/FAQ) упоминать все четыре канала — Telegram, MAX,
  WhatsApp, телефон — без предпочтения «только WhatsApp» или «только звонить»
  ДЛЯ operator, чтобы инвариант бренда не был нарушён на первом релизе.
- Обоснование: каналы — immutable-пункт. Нарушение = регресс бренда ещё до
  запуска.
- Зависимости: REQ-1.3, REQ-3.1.
- Открытые вопросы: реальные аккаунты/номера (@obihod в Telegram, номер
  Wazzup24) — у оператора. Для seed используем placeholder, совместимый с
  форматом (`+7 (495) 000-00-00` уже в существующем seed), реальные значения —
  отдельной US («настройка каналов»).

### REQ-1.6 — Один mock-Case + placeholder-тексты остальных Cases

- Тип: functional
- Источник: intake §«Ответы оператора 2026-04-23» п.1 и п.4;
  `site/content/cases/` (коммит c2f5072).
- MoSCoW: **Must** (один mock), **Won't** в этой US (3–5 реальных кейсов).
- Формулировка: Система ДОЛЖНА после seed содержать ровно 1 mock-Case (арбо ×
  Раменское, существующий из коммита c2f5072), слинкованный через
  `ServiceDistricts.miniCase`, остальные 27 `ServiceDistricts` публикуются без
  `miniCase` и рендерятся с `noindexUntilCase=true` ДЛЯ `seo2`, чтобы
  соблюсти защиту от Scaled Content Abuse, заложенную в `ServiceDistricts`
  beforeValidate-хуке.
- Обоснование: оператор зафиксировал «1 mock, расширение по фактам»; хук
  `requireGatesForPublish` блокирует `publishStatus=published` без `miniCase`,
  поэтому 27 LP остаются в статусе `draft` или `review` + `noindex`, но
  рендерятся, и это единственно корректное поведение в рамках immutable-правила
  «без Scaled Content Abuse» из `ServiceDistricts.ts`.
- Зависимости: REQ-1.2, REQ-2.3.
- Открытые вопросы: нужно ли 27 LP выставлять `publishStatus=draft` или
  `review` — уточнит `sa` вместе с `seo1`. BA-позиция: `draft` + `noindex`,
  пока нет кейса.

### REQ-1.7 — TOV-фильтр: ни одной анти-TOV формулировки

- Тип: business rule (invariant)
- Источник: CLAUDE.md §Immutable («Don't» vocabulary); PROJECT_CONTEXT.md §2;
  хук `.claude/hooks/protect-immutable.sh`.
- MoSCoW: **Must**
- Формулировка: Система ДОЛЖНА в seed-текстах (`intro`, `leadParagraph`,
  `faqGlobal`, `localFaq`, `metaTitle`, `metaDescription`, `bio`, описания
  mock-Case) не содержать формулировки «услуги населению», «имеем честь»,
  «от 1 000 ₽», «в кратчайшие сроки», «индивидуальный подход» (и их вариаций)
  ДЛЯ operator и бренда, чтобы сохранить TOV «порядок под ключ».
- Обоснование: инвариант проекта, хук блокирует правки в `site/content/`.
  Placeholder-тексты легко соскальзывают в казёнщину, если не вписать
  проверку в DoD.
- Зависимости: REQ-1.3, REQ-1.6.
- Открытые вопросы: нет.

### REQ-1.8 — Формат ценников в seed: «фикс-цена за объект»

- Тип: business rule
- Источник: CLAUDE.md §Immutable Do (`«12 800 ₽ за объект»`);
  `Services.ts` (`priceUnit: object|tree|m3|shift|m2`, `priceFrom ≥ 0`,
  `priceTo ≥ 0`).
- MoSCoW: **Must**
- Формулировка: Система ДОЛЖНА в seed-ценах использовать формат «фикс-цена за
  объект/дерево/м³/смену/м²» (поле `priceUnit` из реального enum),
  не выводить формулировку «от N ₽» в `metaTitle`/`metaDescription`
  ДЛЯ operator, чтобы сохранить оффер «фикс-цена за объект».
- Обоснование: «от 1 000 ₽» — прямой анти-TOV пункт. В существующем seed цена
  задана как коридор `priceFrom ... priceTo` (корректно) и как «фикс-цена за
  объект — от 700 ₽ ... до 45 000 ₽» в intro (корректно). Надо сохранить
  этот паттерн во всех 4 кластерах.
- Зависимости: REQ-1.2, REQ-1.7.
- Открытые вопросы: базовые вилки `priceFrom/priceTo` для кластеров «крыши»,
  «мусор», «демонтаж» — передать `cw` + `sa` (в intake-beseda нет). BA-позиция:
  placeholder-вилки от оператора/бригадира, иначе просим `po` запросить у
  оператора (см. §9).

### REQ-2.1 — Блокировка случайного прогона в prod

- Тип: non-functional (safety)
- Источник: intake §«Риски / зависимости» п.2 (`OBIKHOD_SEED_CONFIRM=yes`).
- MoSCoW: **Should**
- Формулировка: Система ДОЛЖНА требовать явного env-флага (например,
  `OBIKHOD_SEED_CONFIRM=yes` + проверку `NODE_ENV` / `DATABASE_URI` на prod-хост)
  перед записью в prod-БД ДЛЯ `do` и operator, чтобы исключить случайный
  прогон seed при CI-деплое.
- Обоснование: идемпотентность не защищает от кейса «seed обновил текст
  placeholder-а поверх ручных правок `cw`». Защита — обязательный подтверждающий
  флаг.
- Зависимости: REQ-1.1.
- Открытые вопросы: финальная форма флага — решение `tamd` + `do`.

### REQ-2.2 — SeoSettings: легальные реквизиты как placeholder

- Тип: functional
- Источник: существующий seed (`taxId: '7847729123'` — СПб, временный);
  MEMORY `project_inn_temp.md` («ИНН временный, не блокировать решения»).

- Формулировка: Система ДОЛЖНА в `SeoSettings` прописать реквизиты уровня

- Зависимости: REQ-1.2.
- Открытые вопросы: использовать ровно текущие placeholder-значения (ИНН

### REQ-2.3 — Фото для LandingPages и Cases: placeholder

- **Д-2.** Структура и helper-ы в `site/scripts/seed.ts` (особенно

  ломает существующий контракт Payload 3 / lexical editor.
- **Д-3.** Существующий mock-Case из коммита c2f5072 доступен в репозитории и
  пригоден для линковки как `ServiceDistricts.miniCase` для пары
  arboristika × ramenskoye.

- **Д-5.** Оператор принимает placeholder-реквизиты SeoSettings (ИНН 7847729123
  временный), согласно MEMORY `project_inn_temp.md`.

## 8. Acceptance Criteria (BA-уровень)

Оператор скажет «приняли», когда:

- **AC-BA-1.** После прогона seed на prod все 28 URL вида
  `https://obikhod.ru/<svc-slug>/<district-slug>/` (4 кластера × 7 пилотных)
  возвращают HTTP 200 (draft-статус LP допустим, но страница рендерится).
- **AC-BA-2.** Прогон seed без env-флага подтверждения на prod-БД завершается
  ошибкой с понятным сообщением и НЕ пишет в БД. С флагом — пишет.
- **AC-BA-3.** Ровно 1 из 28 `ServiceDistricts` (арбо × Раменское) имеет
  `miniCase` и может быть `published`, остальные 27 — в статусе `draft` или
  `review` с `noindexUntilCase=true`, и это зафиксировано в QA-отчёте.
- **AC-BA-4.** Для каждой из 4 `Services` и 7 `Districts` заполнены `metaTitle`
  (в пределах лимита `Services.ts`: ≤60), `metaDescription` (140–160), `h1` /
  `nameNominative` + 4 падежа, Lexical `intro`/leadParagraph — не пустой.
- **AC-BA-5.** `sitemap.xml` на prod содержит как минимум корневые URL услуг
  (4) + районов (7) + опубликованные `ServiceDistricts` (≥1). Отдельно
  `seo2` подтверждает корректность в QA-отчёте.
- **AC-BA-6.** `qa` + `cw` провели поиск по стоп-словам в seed-текстах
  («услуги населению», «имеем честь», «от 1 000 ₽», «в кратчайшие сроки»,
  «индивидуальный подход») — 0 совпадений. Каналы Telegram / MAX / WhatsApp /
  телефон упомянуты в SeoSettings и в ≥1 публичном тексте на каждой из
  4 корневых услуг.
- **AC-BA-7.** Повторный прогон `pnpm seed` (с флагом) на уже прогнанной БД
  завершается с exit 0, выводит читаемый отчёт `созданные: 0 / пропущенные:
  M / обновлённые: 0`, БД не меняется (row count одинаковый до и после).
- **AC-BA-8.** Миграции Payload в рамках US-1 НЕ внесены; backlog-задача
  «Payload migrations» создана в Linear (fills у `po`), связь `blocks: US-1`
  снята.
