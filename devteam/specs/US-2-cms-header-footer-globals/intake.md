## Резюме запроса

Вынести данные Header и Footer из JSX-компонентов
[site/components/marketing/Header.tsx](../../../site/components/marketing/Header.tsx)
и [Footer.tsx](../../../site/components/marketing/Footer.tsx) в **Payload globals**
(1–2 global на решение `sa` + `tamd`), чтобы оператор через `/admin` самостоятельно
редактировал меню, телефон, CTA, слоган, футер-колонки, контакты, соцсети,
реквизиты, копирайт — без релиза кода.

**Без этой задачи:** каждая правка телефона / пункта меню / слогана требует PR и
деплоя → оператор не владеет CMS-ом полноценно → CMS-first позиционирование
проваливается.

**С этой задачей:** Header/Footer становятся контент-артефактом как Services,
Cases, Blog, а не кодом.

## Тип задачи

- **Тип:** Feature (новая возможность в CMS, изменение архитектуры layout'а).
- **Срочность:** `blocker` — в одном пакете с US-1 (seed) перед публичным запуском.
- **Сложность (грубая оценка in, верифицирует sa):** 1–2 дня работы: схема
  global + server-component-читалка + замена JSX на шаблон.

## Открытые вопросы к оператору (до передачи → ba)

1. **Один global или два?** Предпочтительно `sa` + `tamd` решают, но хочется
   услышать интуицию оператора: (а) один global `Navigation` с двумя секциями
   header/footer, (б) два независимых global `Header` и `Footer`. Рекомендация
   `in`: **один global `SiteChrome`** (или `Navigation`) с группами `header`,
   `footer`, `contacts`, `requisites`, `social` — оператор в одном месте
   правит всё, что про «рамку сайта».
2. **Телефон — один или несколько?** Сейчас в шапке `+7 (495) 123-45-67` —
   это финальный рабочий номер Обихода или placeholder, который оператор
   даст после подключения Calltouch? Нужен ли отдельный Telegram/WhatsApp/MAX
   deep-link (`tg://`, `https://wa.me/`, `https://max.ru/`) + форматированный
   phoneDisplay отдельно от tel-ссылки?
3. **Соцсети / мессенджеры в footer:** фиксированные слоты (Telegram, MAX,
   WhatsApp, VK, YouTube, Я.Дзен) или универсальный массив `{type, url}`?
   Рекомендация `in`: **массив** — гибче при добавлении нового канала.
4. **Меню — anchor vs route:** сейчас `#services`, `#calc`… работает только
   на главной. После seed появятся реальные роуты `/uslugi/`,
   `/arboristika/{district}/`. Делаем поля типа «ссылка» (href как строка,
   ответственность оператора писать правильно) или тип с union
   (`{type: 'anchor'|'route', value}`)? Рекомендация `in`: **anchor-friendly
   href string + валидация в админке**.
5. **Реквизиты (ИНН, КПП, ОГРН, юр. адрес):** из memory:
   `project_inn_temp.md` — 7847729123 (СПб) помечен временным. В global класть
   **этот временный** или **оставить поле пустым до получения финальных**?
   Рекомендация `in`: **пустое поле с описанием в админке — «заполнить перед
   публичным запуском»**. Без заглушек.
6. **Сайт-wide баннер** («акция недели», «работаем в праздники», «авария —
   звоните горячую линию»): в этой US или в бэклоге? Рекомендация `in`: в
   **бэклоге** — не раздувать scope P0.
7. **Переключатель региона** (Москва / МО / конкретный район) в header:
   в этой US или в бэклоге? Рекомендация `in`: **бэклог** — после того, как
   ServiceDistricts реально заполнятся в seed и появится навигация по
   районам. До тех пор нет контента, куда переключать.
8. **Копирайт / footer-bottom строка** (© 2026 Обиход · политика
   конфиденциальности · оферта): из поля global или генерится из
   текущего года + фиксированных ссылок? Рекомендация `in`: **год авто,
   ссылки из global**.

## Предварительный бриф для BA

1. **Бизнес-цель:** операционная независимость от разработки. Оператор должен
   редактировать рамку сайта без PR.
2. **Метрика успеха:**
   - Оператор через `/admin` меняет телефон / пункт меню / слоган / реквизит
     за < 30 секунд (без DevTools, без git, без деплоя).
   - После сохранения — изменение на prod появляется за ≤ 60 секунд
     (revalidation / ISR).
   - 0 строк захардкоженных данных в `Header.tsx` / `Footer.tsx` (только
     разметка + чтение из global).
3. **Границы задачи (in предлагает, ba уточняет):**
   - IN: новый global (или два), рефакторинг `Header.tsx` / `Footer.tsx` в
     server-component-читалки, миграция текущих значений из JSX в seed global,
     revalidation при сохранении global.
   - OUT: сайт-wide баннер (вопрос 6), region switcher (вопрос 7) — в backlog;
     мобильное меню / бургер (если сейчас и так работает — не трогать; если
     нет — отдельная UI-задача для `ui` + `fe1/fe2`); смена брендинга лого
     (зона `art`, не этой US).
4. **Риски / зависимости:**
   - Падение/пустой global → Header/Footer должны деградировать **мягко**
     (дефолтные значения в коде или fallback UI, не белый экран).
   - SSG/ISR: global должен инвалидироваться при сохранении — hook на
     `afterChange` у global + `revalidatePath('/', 'layout')` в next.js.
   - Кеш RSC: читаем global один раз за запрос (memoize), не по компоненту.
   - Access: `read: () => true` (публичные данные), `update: admin-only`.
5. **Состав команды (предварительно, финально решает `po`):**
   - `ba` — подтверждение scope и формулировка AC
   - `sa` — спека с US/AC/UML/DoD + решение по 1 vs 2 global
   - `dba` — ревью схемы global (индексы не нужны — единственная запись,
     но фильтр по slug)
   - `tamd` — ADR если оператор хочет нетривиальное (например, переключатель
     региона). Без нетривиала ADR не нужен.
   - `art` + `cw` — контент-ревью итоговых текстов (слоган, меню, CTA) через
     TOV-фильтр
   - `ui` — убеждается что макет не поедет при пустых или длинных значениях
     (max-length в схеме Payload)
   - `be3` или `be4` — схема global + revalidation
   - `fe1` или `fe2` — рефакторинг Header.tsx / Footer.tsx, server-component,
     fallback-состояние
   - `seo2` — проверка что номер телефона tel:-ссылкой + `Organization`
     schema.org читает данные из global (telephone, address, sameAs для
     соцсетей)
   - `qa1` или `qa2` — тест-пасс: edit через админку, изменение на prod,
     mobile-chrome, a11y нав-меню
   - `out` — accept перед переводом в Done

## Ответы оператора (зафиксировано 2026-04-22)

1. **Один global** — `SiteChrome` (или `Navigation`, финальное имя за `sa` и `be3/be4`)
   с секциями `header`, `footer`, `contacts`, `requisites`, `social`.
2. **Телефон — финальный: `+7 (985) 170-51-11`** (т.е. тел-ссылка `tel:+79851705111`,
   display `+7 (985) 170-51-11`). Хранить и phoneDisplay, и phoneE164 отдельно.
   Не placeholder, класть в seed global на старте.
3. **Соцсети — массив `{type, url}`**, `type` из enum: `telegram`, `max`,
   `whatsapp`, `vk`, `youtube`, `yandex-zen`, `other`. Порядок — по drag-and-drop в
   админке.
4. **Меню — вариант «тип + значение» (union, строже).** В админке radio/select:
   - `anchor` → поле `anchor` (строка `services`, `calc`…) → рендер `#<anchor>`
   - `route` → поле `route` (строка `/uslugi/`) → внутренний `<Link>`
   - `external` → поле `url` (полный URL) → `<a target="_blank" rel="noopener">`
   Валидация на уровне Payload + UI-подсказка в админке.
5. **Реквизиты — оставляем временный ИНН `7847729123` (СПб)**. Пометка в админке:
   «временный ИНН, подлежит замене при регистрации юрлица Обиход». Источник:
   `project_inn_temp.md` в user memory. Остальные реквизиты (КПП, ОГРН, юр. адрес)
   — пустые, подпись «заполнить после регистрации юрлица».
6. **Сайт-wide баннер — в бэклог.** Не в этой US.
7. **Region switcher — в бэклог.** Подключаем после seed ServiceDistricts.
8. **Копирайт: год автоматический** (из `new Date().getFullYear()`),
   ссылки «Политика конфиденциальности» / «Оферта» — из global (поля `privacyUrl`,
   `ofertaUrl`), ИНН **не дублировать** в строке копирайта (он уже в блоке
   реквизитов выше).

**Scope US-2 финализирован.** Передаю в работу `ba`.
