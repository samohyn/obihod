## Резюме запроса

Prod-инстанс Payload CMS на VPS 45.153.190.107 имеет схему (залитую одноразово
через `pg_dump --schema-only` из локальной dev БД), но коллекции пустые.
Публичные страницы сайта, построенные на programmatic-SEO (`/arboristika/[district]/`
и аналогичные для остальных 3 кластеров услуг), возвращают 404 — нет данных в
`Services`, `Districts`, `LandingPages`. До залития seed-а публичный запуск и
любой заметный трафик смысла не имеют.

## Тип задачи

- **Тип:** Ops (одноразовая инфраструктурная операция) + Feature (seed-скрипт
  становится активом repo для dev/staging/prod-refresh).
- **Срочность:** `blocker` — блокирует релиз-1 (publicный запуск сайта).
- **Связанные темы:** переход на Payload migrations (см. open question 3),
  seed-стратегия для dev/staging.

## Контекст (что уже было)

- Prod-инфра готова (deployed 2026-04-21): nginx, PM2, Postgres 16, Let's Encrypt,
  backups, pm2-logrotate, security-audit workflow.
- `/api/health` и `/api/health?deep=1` работают — Payload connection up.
- Схема залита (одноразово, `pg_dump --schema-only`).
- В `contex/` есть стратегические артефакты (позиционирование, конкуренты, GTM).
- В `site/content/cases/` есть 1 mock-кейс с фото + страница кейса
  (коммит c2f5072).
- Каталог услуг (50+ позиций) и география (74 района) зафиксированы в
  [PROJECT_CONTEXT.md](../../PROJECT_CONTEXT.md) §4, §5.

## Открытые вопросы к оператору

1. **Источник seed-данных для `Cases`:** есть 1 mock-кейс. Нужны ещё кейсы для
   старта (хотя бы 3–5 в пилотных районах)? Если «да» — чьими силами готовить
   (бригадир + фото, cw + тексты)? Если «нет — начнём с 1, дополним по мере
   выполнения работ» — это тоже валидный путь.
2. **Seed приоритизация:** стартуем с пилотных 7 районов (Одинцово, Красногорск,
   Мытищи, Химки, Истра, Пушкино, Раменское) и всех 50+ услуг → ~350 LandingPages?
   Или берём только топ-4 кластера услуг × 7 районов = 28 LandingPages на MVP?
3. **Переход на Payload migrations** — делать одновременно (US-1 = seed + miграции
   fix) или отдельным US-2? Рекомендация `in`: отдельно, не смешивать риски.
4. **Scope cases-контента для seed** — реальные тексты от `cw` или placeholder?
   Рекомендация `in`: placeholder-текст (структура + заглушки), финальный контент —
   позже отдельной US (`cw` ведёт).

## Предварительный бриф для BA

1. **Бизнес-цель:** разблокировать публичный запуск сайта. Без seed-а SEO-индексация
   и трафик не работают.
2. **Метрика успеха:**
   - После релиза: `/arboristika/odincovo/` → 200 OK (и ещё N programmatic-роутов
     из матрицы cluster × district).
   - Sitemap.xml содержит ≥ N публичных URL.
   - `do` может выставить сайт в Яндекс.Вебмастер и Google Search Console на
     индексацию.
3. **Границы задачи (из спора с оператором):**
   - IN: seed Services, Districts, LandingPages (в стартовом объёме, согласованном
     в AC от `sa`), 1+ Cases, Prices coefficients для калькуляторов, FAQ.
   - OUT: миграции Payload (отдельная US), реальные тексты от `cw` (отдельные
     US), фото реальных работ (отдельная инициатива через бригадира).
4. **Риски / зависимости:**
   - Скрипт должен быть идемпотентным — защита от повторного запуска.
   - Prod-запуск seed — через env-флаг (`OBIKHOD_SEED_CONFIRM=yes`) или ручной
     шаг, не в каждом деплое.
   - Если `dba` скажет «нужны ещё индексы перед seed-ом больших таблиц» — следуем.
5. **Состав команды (предварительно, финально решает `po`):**
   - `ba` — бизнес-декомпозиция и scope (MVP vs Full)
   - `sa` — спека с AC и DoD
   - `dba` — ревью схемы/индексов, план seed-запуска
   - `be3` или `be4` — скрипт `site/scripts/seed.ts`
   - `seo1` — приоритизация районов (пилот vs wave-2), рекомендация по
     title/meta для seed LandingPages
   - `do` — one-off команда в deploy.yml или ручной запуск через SSH
   - `qa1` или `qa2` — тест-пасс после запуска (страницы 200, sitemap OK)
   - `out` — accept перед переводом в Done
