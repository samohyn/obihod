## 0. Ключевой UX-принцип (сквозной через все экраны)

> **«Никакого экрана без подсказки. Каждое поле отвечает на вопрос оператора `что сюда и зачем` в 1 строку. Каждый список — на вопрос `с чего начать`. Каждая ошибка — на вопрос `что мне сейчас сделать`.»**

Всё остальное — производные этого принципа:
- Dashboard — не список коллекций, а список **задач-сценариев**.
- List view — не «No results», а **CTA с контекстом**.
- Edit view — не голая форма, а **форма + inline-help + preview-кнопка в видимой зоне**.
- Ошибка publish-gate — не stacktrace, а **человеческое «добавь блок X, потому что Y», с якорем на место**.
- Docs — не внешний wiki, а **вкладка в sidebar**.

Этот принцип — фильтр для всех последующих UX-решений в Обиходе (не только US-3).

### 1.5 Микровзаимодействия

- Hover по tile: поднимается тень (`box-shadow` +4 px), cursor pointer, рамка уплотняется (`border-color: --accent`).
- Focus (keyboard): outline 2 px `--accent` + слегка поднятый фон `#efebe0`.
- Click: мгновенный переход; небольшая задержка (opacity 0 → 1 на новом экране) не нужна — Payload route transition хватит.
- Widget строки — `hover: background: #efebe0`, cursor pointer.

### 1.6 A11y-требования

- **Landmark'и:** `<header>`, `<nav aria-label="Главное меню">`, `<main aria-label="Начало работы">`, `<aside aria-label="Боковые виджеты">`.
- **Tile'ы:** `<a href="...">` с `aria-label="Добавить район — 3 шага, около 5 минут"` (полный текст для screen reader; визуально разбит на строки). Не используем `<div onClick>`.
- **Порядок Tab:** header-menu → sidebar (↑/↓ между коллекциями) → 6 tile'ов (слева-направо, сверху-вниз) → widget «Открытые заявки» → widget «Недавно».
- **Screen reader labels:** greeting — `<h1>` (единственный на странице), «С чего начать» — `<h2>`, widget-заголовки — `<h2>`. Порядок H-level соблюдаем.
- **Hit area tile:** минимум 180 × 120 px, полностью кликабельно (не только текст).
- **Контраст:** проверен `art-concept §8`, внутри tile'а фон `--surface` `#ffffff` на `--background` `#f7f5f0`, текст `--text` `#1c1c1c` — 16.8:1 AAA.
- **Reduced motion:** hover-анимация и shimmer отключаются при `prefers-reduced-motion: reduce`.

### 2.7 A11y-требования

- Таблица: нативный `<table>` с `<th scope="col">`, `<caption class="sr-only">Список услуг</caption>`.
- Sort по колонке: `<th>` с `aria-sort="ascending|descending|none"`, клик — keyboard-accessible.
- Фильтры и поиск: `<label>` связан с input (визуально скрыт через `sr-only`, если экран плотный).
- Row-action dropdown: `<button aria-haspopup="menu" aria-expanded="..." aria-controls="menu-id">`. Menu — Radix UI или аналог, keyboard-ready.
- Модалка «Дублировать»: `role="dialog"` + `aria-labelledby` + **focus trap** + Esc закрывает. Фокус возвращается на кнопку `⋯`, с которой модалку открыли.

| Состояние | Что показываем |
|---|---|

### 3.5 Микровзаимодействия drag-drop

- **Pickup:** mousedown на drag-handle → через 150 ms (или сразу, если сдвиг > 4 px) блок становится «поднятым» (scale 1.02, shadow 16 px, opacity 0.85).
- **During drag:** остальные блоки плавно сдвигаются (CSS transform transition 200 ms), drop-zone между блоками подсвечивается рамкой `--accent` 2 px dashed.
- **Drop:** блок встаёт на место (scale 1, shadow 4 px), порядок меняется мгновенно, autosave тикнет через 2 с.
- **Cancel (Esc во время drag):** блок возвращается на исходную позицию.

### 3.6 A11y-требования — САМАЯ СЛОЖНАЯ ЧАСТЬ

Drag-drop клавиатурой — критично по WCAG 2.2 AA (2.1.1 Keyboard, 2.5.7 Dragging Movements). Payload `blocks` field в 3.x поддерживает keyboard-reorder, **нужно верифицировать `be4` что работает**. Если нет — обязаны дописать.

| Требование | Механика |
|---|---|
| **Keyboard-reorder** | Tab до блока → `Space`/`Enter` на drag-handle → режим reorder активен. `↑`/`↓` двигают блок вверх/вниз. `Space`/`Enter` — закрепить позицию. `Esc` — отменить, вернуть на исходную. |
| **Screen-reader announcements** | При pickup: `aria-live="assertive"` — «Блок 3 "MiniCase" поднят. Используй стрелки для перемещения». При move: «Блок 3 перемещён на позицию 2». При drop: «Блок закреплён на позиции 2 из 5». При cancel: «Перемещение отменено». |
| **Focus management** | После reorder фокус остаётся на drag-handle того же блока (теперь на новой позиции). После add — фокус перескакивает на только что добавленный блок (его expand-toggle). После delete — фокус на следующий блок (или на кнопку «+ Добавить блок», если удалён последний). |
| **Visible focus ring** | Все интерактивные элементы (handle, `⋯`, expand, inputs) — `:focus-visible` outline 2 px `--accent` + offset 2 px. |
| **`aria-roledescription="Блок страницы"`** | На каждом блоке — для screen reader. |
| **Menu `⋯`** | `role="menu"`, элементы `role="menuitem"`. Стрелки ↑/↓ переключают, Esc закрывает, Tab выходит. |
| **Labels inputs внутри блока** | Все поля — нативный `<label>`. `admin.description` выводится как `<span id="desc-..."` связанный через `aria-describedby`. |

### 3.7 Risk R10 — performance на 10+ блоках

- **Collapsed by default** (Payload `admin.initCollapsed: true`) — обязательно.
- **Lazy-render развёрнутого состояния:** когда блок collapsed, его внутренние поля не монтируются в DOM (реализация — `be4`, возможно через `React.lazy` / condition на render).
- **Virtualized list** при 20+ блоках — обсудить с `fe1` (react-window внутри Payload admin — может конфликтовать с D&D; скорее всего, не нужен, max 15 блоков на страницу).
- **Benchmark 15-блочная страница** — release-gate от `qa2` (PO-заявил в PO Review §7 RISK-PO-2).

### 5.4 A11y-требования

### 9.1 CJM-1 — «Добавить район с нуля» (самый частый)

**Персона:** оператор Георгий (P-OP — единственный `admin`). Контекст: развёртывание programmatic-сетки US-4 (60+ районов). Сценарий повторяется N раз, поэтому каждая секунда × N.

| 3 | Видит форму ServiceDistrict | Edit view | «Что это за поля? Что обязательно?» | Тревога | Голая форма без описаний | §5.2 inline-help под каждым полем + §5.1 баннер сверху |
| 4 | Выбирает услугу и район | Relationship-селект | «Вижу dropdown, выбираю» | Норм | — | Payload-native, OK |

| 11 | Видит, что опубликовано | Sticky-bar статус | «Проверю на prod» | Спокойно | — | — |
| 12 | Хочет создать следующий район | Возврат к Dashboard? | «6 раз идти в Dashboard = скучно» | Лёгкая усталость | — | **REQ-2.5 row-action «Дублировать в район»** в ServiceDistricts list view (§2.6) |

**Критическая точка:** шаг 3 (первый раз вижу форму). Без inline-help оператор бросит.
**Ускоритель цикла:** шаг 12 — `Duplicate-to-district` превращает «N × 8 минут» в «N × 3 минуты».

**Время сценария:**
- **Сегодня (без US-3):** ≈ 10-15 мин на первый район + ещё столько же «попытка понять, что не так при publish».
- **После US-3 (первый район):** ≈ 5-6 мин (за счёт inline-help, preview, publish-gate с понятной ошибкой).
- **После US-3 (следующие N-1 районов с duplicate):** ≈ 2-3 мин на район.

### 9.2 CJM-2 — «Собрать программатик-страницу из блоков» (самый новый)

**Персона:** оператор. Контекст: оператор впервые столкнулся с блочным редактором после US-5 (библиотека блоков готова). Нужна programmatic-LP «Крыши × Мытищи». Сценарий одноразово сложный, потом становится рутиной.

| # | Шаг | Touchpoint | Мысль | Эмоция | Требование к UX |
|---:|---|---|---|---|---|
| 1 | На Dashboard видит tile «Собрать страницу из блоков» | §1 | «А как это вообще работает?» | Неуверенно | Tile ведёт **в гайд**, не в пустую форму — §1.3 |
| 2 | Читает гайд `/admin/docs/build-page-from-blocks` | §6 docs-article | «Ага, каждый блок — кусочек страницы» | Осмысление | Гайд со скриншотами, чеклистом |
| 3 | Кликает `[ Создать страницу → ]` из гайда | LandingPage create | Новая форма | Готовность | Preset: пустой blocks[] |
| 4 | Видит панель blocks[] пустую + empty-state | §3.4 empty blocks | «Что добавить первым?» | Пауза | Empty-state: «Добавь Hero или text-content» + CTA |
| 5 | Кликает «+ Добавить первый блок» → палитра | §4 | «Сколько вариантов!» | Вау | Карточная сетка + категории + фильтр |
| 6 | Выбирает Hero → блок добавлен, раскрыт | §3.2 | «Красиво» | Удовлетворение | — |
| 7 | Заполняет Hero, сворачивает | — | «Следующий» | Потоковое состояние | Collapse с превью thumbnail'а |
| 8 | Добавляет ещё 5 блоков: text-content, FAQ, cases, testimonials, lead-form | §4 | «Быстро» | Уверенно | Inline «+» между блоками — без модалки |
| 9 | Перетаскивает FAQ выше cases | §3.5 drag | «Удобно» | Довольство | D&D работает mouse + keyboard |
| 10 | Жмёт Preview | §7 | «Гляну» | Интерес | Preview в новой вкладке с баннером |
| 11 | В preview видит «Пустой блок FAQ» | — | «Забыл заполнить» | Досада | Preview показывает как есть — **не скрываем пустые блоки**, иначе сюрприз на prod |
| 12 | Возвращается в admin, заполняет FAQ | §3.2 | «Исправил» | Удовлетворение | — |
| 13 | Publish | §7 или main | «Опубликовано» | Гордость | — |

**Критические точки:**
- Шаг 1-2: решение **вести в гайд, а не в форму**. Это контр-интуитивно, но оправдано — без понимания концепции блоков оператор соберёт кашу (см. R11 thin-страницы).
- Шаг 4: empty blocks — **самое важное место** для CTA. Не «No blocks», а «С чего начать».
- Шаг 11: pre-publish проверка через preview. UX должен показывать **все** блоки, включая пустые, — иначе publish-gate сюрприз.

**Время сценария:**
- **Первый раз (новый паттерн):** 15-20 мин (включая чтение гайда).
- **Следующие страницы:** 8-10 мин (AC-BA-12 target).

### 9.3 CJM-3 — «Опубликовать → publish-gate ошибка → исправить → опубликовать» (ошибочный)

**Персона:** оператор. Контекст: собрал страницу из 3 блоков (Hero + MiniCase + FAQ), забыл text-content и lead-form. Жмёт publish.

| # | Шаг | Touchpoint | Мысль | Эмоция | Что происходит |
|---:|---|---|---|---|---|
| 1 | Собрал 3 блока в LandingPage | §3 | «Готово» | Уверенно | — |
| 2 | Жмёт «Опубликовать» | Sticky-bar | — | Ожидание | — |
| 3 | Видит красный/янтарный баннер `⚠ Нельзя опубликовать` | §8.1 | «Ой» | Раздражение (первая секунда) | `aria-live` объявляет screen reader'у |
| 4 | Читает список нарушений | §8.1 | «А, мне нужны форма и текст 300 слов» | Понимание | Список — не stacktrace, а действия |
| 5 | Жмёт `[ Добавить блок ]` для lead-form | §4 | «Это про это» | Уверенно | Палитра открывается с префильтром «Формы» |
| 6 | Выбирает Lead-form → вставлен | §4 | «Ещё text-content» | Потоковое состояние | Баннер наверху обновляется — одна строка нарушения исчезла |
| 7 | Добавляет text-content блок | §4 | — | — | Баннер показывает «300 слов нужно, сейчас 0» |
| 8 | Начинает писать в text-content | Lexical | — | — | Счётчик слов в баннере live-обновляется: «50 / 300» |
| 9 | Дописал до 320 слов | — | «Хватит» | Удовлетворение | Баннер схлопывается, Publish становится enabled |
| 10 | Жмёт Publish | — | «Готово» | Триумф | Publish проходит |

**Критические точки:**
- Шаг 3: **текст баннера** решает всё. «Validation failed: at least one CTA block required» vs «Нужна форма заявки или CTA-баннер — добавь блок». Первое — стыд, второе — действие.
- Шаг 8: **live-обновление** баннера (не перепроверка после submit) — разница между «раздражает» и «помогает».
- Шаг 9: **визуальная награда** (fade-out, glow на Publish) — подкрепление «ты молодец, исправил».

**Время сценария:** ≈ 3-5 мин после первого publish-click до финальной публикации. **Без US-3:** оператор пишет в чат разработчику «почему не публикует?», цикл — часы.

## 11. 2 CJM-боли для `sa` (в NFR)

Эти боли **вскрылись при прокладке CJM** выше, выходят за scope REQ-1.*/3.*/6.*, но `sa` должен учесть их в `sa.md` как NFR или адресовать в backlog.

### 11.1 Боль — Upload медиа медленный (CJM-1 шаг 6)

- **Контекст:** при добавлении района оператор грузит 3+ фото (MiniCase). Сегодня upload через Payload + Beget S3 — заметная пауза (2-4 сек на файл).
- **UX-следствие:** в потоке «добавить 60 районов за спринт» это накапливается в десятки минут ожидания.
- **NFR-требование для `sa`:**
  - target: upload одного фото ≤ 1.5 сек на файл до 2 МБ.
  - механики: параллельный upload (не sequential), оптимистичный UI (показать placeholder мгновенно, закоммитить позже), progress bar per файл.
- **Зона:** возможно US-3 (если легко), иначе backlog к Media-коллекции.

### 11.2 Боль — Отсутствие «последних значений» при создании похожей сущности (CJM-1 шаг 12, CJM-2 шаг 8)

- **Контекст:** оператор создаёт много однотипных записей (ServiceDistricts × 60 районов, LandingPages × 4 услуги × 15 районов). Каждое создание — это 80% тех же значений + 20% свежих.
- **UX-следствие:** REQ-2.5 (duplicate-to-district) частично решает для ServDistricts. Но для LandingPages и других массовых коллекций нет паттерна.
- **NFR-требование для `sa`:** архитектурно рассмотреть паттерн «шаблоны документов» или «создать как копию» на уровне **всех коллекций с массовым созданием** (не только ServDistricts). Не реализация в US-3, а **решение паттерна** в `sa.md` и задача в backlog.
- **Зона:** US-5/US-6 (library blocks + programmatic-сетка — естественно требуют этого паттерна).
